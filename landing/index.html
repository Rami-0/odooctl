Why odooctl

The operational lifecycle generic deploy tools miss

Odoo isn't a stateless web app. Backups must include the filestore, staging clones must not email real customers, and module updates can brick a database. odooctl is built around exactly those problems.

⛑

Safety by default

Deploys to protected environments take a database + filestore backup first. Failed module updates and failed health checks stop the deployment with a non-zero exit.

✓

Verify before destroy

Restores land in a temporary database and are only swapped into place after they succeed — a bad backup never destroys a working environment.

⎘

Staging clones you can trust

Sanitized by default: mail servers, crons, payment providers, queue jobs, OAuth secrets, IAP tokens, webhooks — and Odoo 19 passkeys — all neutralized.

⇣

Adopt what you already run

odooctl import detects an existing compose deployment read-only, previews the generated config, and only writes on --yes — then runs preflight checks and takes a safety backup.

⟲

Upgrade rehearsal, not roulette

migrate rehearse clones production into a throwaway database, runs the OpenUpgrade-backed upgrade there, health-checks, and writes a report. Production is never touched.

⚿

Secrets stay secret

Config references secrets by environment-variable name only. Logs, errors, and streamed operation events are redacted; passwords never appear on process argv.

▦

Local API & web UI, RBAC'd

odooctl serve runs a localhost-only REST API plus a no-build web UI with viewer/operator/admin/owner roles and a privileged runner in a separate process.

⛨

Protected environments

Production-tier environments require elevated confirmation for destructive operations — enforced identically in the CLI, the API, and the web UI.

⚕

Doctor & disaster recovery

Preflight checks, health diagnostics, restore points, and a documented DR runbook — because 3 a.m. is the wrong time to improvise.

Quickstart

Two ways in — adopt or start fresh

Either way, you keep your server, your data, and your compose files.

A. Adopt an existing Compose deployment

Read-only detection first; nothing is written without --yes.

# Detect and preview — no containers touched
$ odooctl import /srv/odoo/acme

# Adopt: write config, run doctor, take a safety backup
$ odooctl import /srv/odoo/acme --name acme --yes

# First verified backup, then a sanitized staging env
$ odooctl -p acme backup production --verify
$ odooctl -p acme env create staging --clone-from production

B. Start a fresh project

Scaffolds from a pinned stack template; secrets by env-var name, never inlined.

$ mkdir acme && cd acme
$ odooctl setup --name acme --stack odoo-19-community

# Edit domain + compose file, then:
$ export ODOO_DB_PASSWORD='a-strong-secret'
$ odooctl validate && odooctl doctor
$ odooctl deploy production --branch main
