Open-source Odoo.sh for your own server.

odooctl is a CLI-first, Odoo-aware control plane for self-hosted Odoo on Docker Compose. Verified backups, sanitized staging clones, safe deploys, rollback, and upgrade rehearsal — on your infrastructure.

Why odooctl

The operational lifecycle generic deploy tools miss.

Odoo isn't a stateless web app. Backups must include the filestore, staging clones must not email real customers, and module updates can brick a database.

Safety by default

Deploys to protected environments take a database and filestore backup first. Failed updates and health checks stop the deploy with a non-zero exit.

Verify before destroy

Restores land in a temporary database and are only swapped in after they succeed — a bad backup never destroys a working environment.

Sanitized staging clones

Mail servers, crons, payment providers, queue jobs, OAuth secrets, IAP tokens, webhooks — and Odoo 19 passkeys — neutralized by default.

Adopt what you run

odooctl import detects an existing compose deployment read-only, previews the config, and only writes on --yes.

Upgrade rehearsal

migrate rehearse runs the OpenUpgrade-backed upgrade in a throwaway database and writes a report. Production is never touched.

Secrets stay secret

Config references secrets by env-var name only. Logs, errors, and operation events are redacted; passwords never appear on argv.

Local API & web UI

odooctl serve runs a localhost-only REST API and web UI with viewer / operator / admin / owner roles and an isolated privileged runner.

Protected environments

Production-tier environments require elevated confirmation for destructive operations — in the CLI, API, and web UI alike.

Doctor & disaster recovery

Preflight checks, health diagnostics, restore points, and a documented DR runbook — because 3 a.m. is the wrong time to improvise.

Quickstart

Adopt an existing stack, or start fresh.

Either way, you keep your server, your data, and your compose files.

Adopt an existing Compose deployment

Read-only detection first; nothing is written without --yes.

# Detect and preview — nothing touched
$ odooctl import /srv/odoo/acme

# Adopt: config, doctor, safety backup
$ odooctl import /srv/odoo/acme --name acme --yes

# Verified backup, sanitized staging
$ odooctl -p acme backup production --verify
$ odooctl -p acme env create staging --clone-from production

Start a fresh project

Pinned stack templates; secrets by env-var name, never inlined.

$ mkdir acme && cd acme
$ odooctl setup --name acme --stack odoo-19-community

# Edit the domain and docker-compose.yml, then:
$ export ODOO_DB_PASSWORD='a-strong-secret'
$ odooctl validate && odooctl doctor
$ odooctl deploy production --branch main
